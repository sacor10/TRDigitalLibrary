# TR Digital Library

> A scholarly, accessible, open archive of Theodore Roosevelt's complete works and correspondence.

This is a working **proof-of-concept**: a typed monorepo that ingests public-domain TR
texts from remote sources at seed time, indexes them in SQLite FTS5, exposes a
documented REST API, and renders a dual-pane (facsimile + transcription) reader with
search, browse, and timeline views.

The goal is not to ship every feature тАФ it is to prove the architecture so that the
production build can extend it confidently.

---

## 1. Project overview

### Vision
Make Theodore Roosevelt's complete corpus тАФ speeches, letters, diaries, articles,
autobiography тАФ readable, searchable, citable, and **freely linkable** to the
canonical public-domain record. Combine the rigor of a scholarly edition with the
accessibility of a modern web app.

### Audience
- **Researchers and historians** тАФ full-text search, provenance, citations, an
  OpenAPI-documented dataset they can mine, draft-vs-final diffing, scholarly
  annotations.
- **Educators and students** тАФ a clean reader, contextual essays, a chronological
  timeline, citation export in three styles.
- **Curious enthusiasts** тАФ a browseable, beautiful library that works on a
  phone, on a screen reader, in dark mode, offline-first where possible.

### Guiding principles
1. **Open access.** Code is MIT. Content is fetched at seed time from public-domain
   sources and never re-bundled тАФ the canonical URL is always one click away.
2. **Scholarly rigor.** Every document carries provenance, source attribution, a
   stable id, and the structured metadata needed to cite it. Annotations and
   corrections are first-class.
3. **Accessibility first.** Semantic HTML, keyboard-only flows, ARIA where it
   helps, dark mode, focus rings, color contrast тАФ not an afterthought.
4. **Progressive scale.** SQLite + FTS5 today, Postgres + Meilisearch + IIIF
   tomorrow, with the same shared schemas.

## 2. Quick start

```bash
# 1. install (npm workspaces handles the three packages)
npm install

# 2. seed the SQLite database тАФ fetches transcriptions from Wikisource
#    and caches them locally. Requires outbound HTTPS.
npm run seed

# 3. start both server (:3001) and client (:5173) in watch mode
npm run dev

# tests
npm run test          # all workspaces
npm run test -w server
npm run test -w client

# production build (type-check + vite build)
npm run build
```

Open <http://localhost:5173> in a browser. The OpenAPI spec is at
<http://localhost:3001/api/openapi.json>.

> **Offline note:** if the seed step cannot reach Wikisource, every document is
> still inserted with full metadata and a `sourceUrl` link. Search will match on
> titles, recipients, and tags only until you re-run `npm run seed` from a
> connected environment.

### Ingesting TEI documents

Validate and ingest a folder of TEI/XML documents into the library:

```bash
npm run ingest-tei -- /path/to/tei-folder            # validate, normalize, insert
npm run ingest-tei -- /path/to/tei-folder --dry-run  # parse + validate only
npm run ingest-tei -- /path/to/tei-folder -r         # recurse into subfolders
npm run ingest-tei -- /path/to/tei-folder --db ./data/library.db
```

Each `.xml` file is checked for well-formedness and required TEI structure
(`teiHeader`, `fileDesc`, `titleStmt/title`, `publicationStmt`, `sourceDesc`,
`text/body`). Valid documents are upserted into `documents` (with the original
TEI preserved in `tei_xml`) and their structural hierarchy тАФ `div`, `p`, `lg`,
`l`, `quote`, `list`, `item`, `head`, `note` тАФ is unrolled into
`document_sections` so each section is independently queryable and
FTS5-searchable. The CLI prints a per-file report and exits non-zero if any
file fails validation.

### Stack rationale

| Layer       | POC choice                             | Why                                                                                                           |
| ----------- | -------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| Schemas     | Zod in `/shared`                       | Single source of truth; validates at both API boundaries; OpenAPI generated for free.                         |
| Server      | Express + better-sqlite3               | Sync SQLite is perfect for FTS5; Express is boring and well-understood.                                       |
| Search      | SQLite FTS5 (`porter unicode61`, BM25) | Zero ops, real ranking, real snippets. Trivially upgradable to Meilisearch with the same shared schemas.      |
| API surface | OpenAPI 3.1 from Zod                   | Spec drives doc generation, client codegen, and contract testing.                                             |
| Client      | React 18 + Vite + Tailwind + RQ        | Fastest path to a polished, accessible reader. RQ memoizes `/search` and `/documents` calls for free.         |
| Tests       | Vitest + supertest                     | Same runner client and server. Supertest hits real Express; in-memory SQLite per test suite for hermeticism. |

### POC тЖТ production migration

| Concern             | POC                          | Production                                                       |
| ------------------- | ---------------------------- | ---------------------------------------------------------------- |
| RDBMS               | SQLite (better-sqlite3)      | Postgres (pg / drizzle / kysely)                                 |
| Full-text search    | SQLite FTS5                  | Meilisearch (or Elasticsearch); same query API at the API layer  |
| Image hosting       | Remote LoC / Wikimedia URLs  | Internet Archive IIIF (`iiif.archive.org`) тАФ $0 for public-domain; or serverless-iiif (AWS Lambda + S3) if dynamic transforms needed |
| Transcription store | Cached at seed time          | TEI/XML primary store; HTML/plain text projections               |
| Auth                | none                         | OAuth (researcher accounts), JWT for API rate-limit tiers        |
| Deploy              | Local `npm run dev`          | Docker тЖТ Fly.io / Railway тЖТ AWS ECS as scale demands              |

## 4. Roadmap тАФ five pillars

Legend: **S** = small (тЙд1 day) ┬╖ **M** = medium (1тАУ3 days) ┬╖ **L** = large (1тАУ2 weeks) ┬╖ **XL** = epic (multi-sprint)

### Pillar 1 тАФ Comprehensive Collection

- [x] **Metadata-driven seed pipeline (URL-only)** тАФ S тАФ done in POC
- [x] **TEI/XML ingestion pipeline** тАФ XL тАФ implemented with `fast-xml-parser` (pure Node, no Saxon/tei-publisher runtime required). See `server/src/ingest/` and `npm run ingest-tei -- <folder>`. Validates well-formedness and required TEI structure, preserves hierarchy in `document_sections`, retains raw TEI in `documents.tei_xml`.
- [ ] **IIIF image server integration** тАФ L тАФ Internet Archive IIIF ($0, public-domain content only). Upload facsimiles to archive.org; endpoints auto-generated at `iiif.archive.org`; no server, no config, no egress cost; full IIIF Image API 3.0; no SLA or manifest control. Acceptance: every facsimile served via IIIF Image API 3.0; deep-zoom in viewer.
- [ ] **OCR pipeline (Tesseract тЖТ Transkribus)** тАФ XL тАФ Tesseract free; Transkribus credits ~тВм0.05/page for handwriting. Acceptance: a typewritten page yields тЙе98% character accuracy; a handwritten letter yields тЙе90% via Transkribus.
- [ ] **Provenance tracking** тАФ M тАФ Acceptance: every field on every document records its origin URL, fetched-at timestamp, and editor identity for any subsequent corrections.
- [ ] **Multi-format exports (PDF / EPUB / TEI)** тАФ L тАФ Acceptance: per-document export buttons; PDF preserves typography; EPUB validates with `epubcheck`; TEI validates against P5.
- [ ] **Letters of Theodore Roosevelt corpus (~150,000 items)** тАФ XL тАФ depends on partnership with the Theodore Roosevelt Center. Acceptance: ingestion of the full Morison/Blum Harvard edition with structured recipients and dates.

### Pillar 2 тАФ Intelligent Search

- [x] **FTS5 with snippet highlighting and BM25 ranking** тАФ done in POC
- [ ] **Migrate search to Meilisearch** тАФ M тАФ self-hosted free, or Meilisearch Cloud ($30+/mo). Acceptance: same `/api/search` contract, typo tolerance, faceted filters return in <100ms at p99 over the full corpus.
- [ ] **Semantic search via embeddings** тАФ L тАФ OpenAI `text-embedding-3-small` (~$0.02/1M tokens) or local `bge-small-en` + pgvector. Acceptance: `/api/search?semantic=1` returns conceptually-related results not lexically matched.
- [ ] **Network graph of correspondents** тАФ L тАФ D3 or Cytoscape.js. Acceptance: clicking a correspondent shows their letters and the network of people TR connected them to.
- [ ] **Topic modeling (BERTopic)** тАФ L тАФ Acceptance: cluster the corpus into 30тАУ60 themes; surface drift over time.
- [ ] **Sentiment analysis** тАФ M тАФ Acceptance: per-document polarity score; "TR's mood across the 1912 campaign" as a demo chart.

### Pillar 3 тАФ Scholarly Apparatus

- [ ] **Annotations system (W3C Web Annotations)** тАФ L тАФ Acceptance: any user can highlight a passage; logged-in scholars can attach notes; notes are referenceable by URL and exportable as JSON-LD.
- [ ] **Expert essay CMS** тАФ M тАФ MDX in repo or Sanity ($99+/mo). Acceptance: subject-matter experts can publish contextual essays linked to specific documents.
- [ ] **Cross-reference linking engine** тАФ L тАФ Acceptance: when document A mentions document B by date or correspondent, a sidebar link appears automatically.
- [ ] **Draft-vs-final diff viewer** тАФ M тАФ `diff-match-patch`. Acceptance: any document with multiple versions shows a side-by-side highlighted diff.
- [ ] **Biographical timeline data model** тАФ M тАФ Acceptance: a normalized events table (born, married, elected, тАж) renders alongside the document timeline.

### Pillar 4 тАФ Accessibility & Open Access

- [x] **Skip-to-content, ARIA tablist, focus rings, dark mode** тАФ POC baseline
- [ ] **WCAG 2.2 AA audit** тАФ M тАФ Axe + Pa11y in CI; manual screen-reader pass with NVDA + VoiceOver. Acceptance: zero serious violations.
- [ ] **Screen reader testing** тАФ S тАФ Acceptance: every page is fully usable in NVDA, JAWS, and VoiceOver.
- [ ] **Text-to-speech via Web Speech API** тАФ S тАФ Acceptance: a "Read aloud" button reads the transcription in the user's preferred voice.
- [ ] **Translations via i18next** тАФ L тАФ community-sourced. Acceptance: at minimum English + Spanish UI; document-level translations contributed via PR.
- [ ] **DOI minting via DataCite** тАФ M тАФ DataCite membership ~$2,500/yr. Acceptance: every document has a citable DOI resolved to its persistent URL.
- [ ] **Persistent URLs** тАФ S тАФ Acceptance: documented URL contract; `/documents/:id` never breaks across refactors.
- [ ] **User accounts for saved collections** тАФ L тАФ Auth.js + Postgres. Acceptance: users can save, tag, and share document collections.

### Pillar 5 тАФ Interactive & Community

- [ ] **Public REST + GraphQL APIs with rate limiting** тАФ M тАФ Acceptance: REST already exists; GraphQL endpoint at `/api/graphql`; default 60 req/min per IP, 1k for keyed clients.
- [ ] **Geographic correspondence map (Leaflet/Mapbox)** тАФ M тАФ Mapbox tier ~$0+/mo at low volume. Acceptance: pins for every recipient location; clicking opens letters.
- [ ] **Word frequency visualizations** тАФ S тАФ Acceptance: per-document and corpus-wide word clouds, filterable by date range.
- [ ] **Moderated annotation system** тАФ M тАФ Acceptance: annotation queue with admin UI; spam detection.
- [ ] **Contributor verification** тАФ L тАФ ORCID OAuth. Acceptance: scholars sign in with ORCID and contributions are signed.
- [ ] **Audio readings (volunteer-recorded)** тАФ L тАФ Acceptance: per-document audio clips streamed from S3 with transcript synchronization.

## 6. Contributing

### Code style

- TypeScript strict mode everywhere (`noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`).
- Functional React components with hooks.
- Named exports preferred.
- ESLint + Prettier enforced; run `npm run format` and `npm run lint`.
- Conventional commits: `feat(client): тАж`, `fix(server): тАж`, `docs: тАж`, etc.

### PR process

1. Branch from `main`: `feat/<short-name>`.
2. Write or update tests alongside the change (Vitest).
3. Ensure `npm run lint`, `npm run test`, and `npm run build` all pass.
4. Open a PR; the title should be a conventional commit; the body describes
   intent, screenshots for UI changes, and any data-model migrations.

### How scholars submit transcription corrections

1. Open an issue using the "Transcription correction" template.
2. Identify the document by its `id` (visible in the URL: `/documents/<id>`).
3. Cite the canonical source (LoC, TR Center, Houghton).
4. A maintainer or contributor opens a PR amending `data/seed.json` (metadata
   fixes) or тАФ once the annotation system lands тАФ a moderated correction
   record.
